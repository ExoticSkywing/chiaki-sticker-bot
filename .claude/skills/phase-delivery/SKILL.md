# 阶段性交付包 Skill

## 触发场景

当用户提出长期项目、大项目、阶段性需求、交付、验收、接棒、交接、handoff、下一团队继续开发等需求时，使用本 skill。

更强触发：当用户出现以下语义时，视为上一阶段已验收，必须立即收口：

- “满意”；
- “可以了”；
- “验证通过”；
- “先把前面的落盘”；
- “在加之前，我对之前所做的感到满意”；
- 其他明确表示当前阶段已经达到预期的说法。

如果用户一边表达满意、一边提出下一步需求，顺序必须是：

1. 先生成 / 补齐上一阶段交付包；
2. 更新 `docs/HANDOFF.md`；
3. 再继续下一阶段需求。

## 接手规则

新 agent / 新会话接手非 quick task 时，必须先读：

1. `docs/HANDOFF.md`；
2. 最新的 `docs/phase-deliveries/*.md`，如果已经存在。

不要只根据当前聊天片段继续推进长期项目。

## 核心契约

```text
甲方不满意：继续推进
甲方满意：生成阶段性交付包 + 更新项目接棒入口 + 再继续下一阶段
```

## 未验收时

用户仍不满意，或需求尚未阶段性满足时：

1. 继续理解问题、推进方案、实现或修正；
2. 执行自己力所能及、可稳定运行的验证；
3. 遇到人类更适合的验证，明确交给人类；
4. 不把中间过程包装成正式交付物。

## 已验收时

用户明确满意、说“可以了”、说“进入下一阶段”、或表示阶段性需求满足时：

1. 停止继续堆实现；
2. 在 `docs/phase-deliveries/` 生成阶段性交付包；
3. 更新 `docs/HANDOFF.md`；
4. 最终汇报交付包路径、接棒入口路径、验证结果和后续建议。

可用命令：

```bash
lpd status
lpd close --title "<阶段名>" --summary "<验收结论>"
```

如果没有安装全局命令，使用：

```bash
npx --yes github:ExoticSkywing/loop-phase-delivery status
npx --yes github:ExoticSkywing/loop-phase-delivery close --title "<阶段名>"
```

`close` 只负责生成正确文件和更新索引；agent 仍必须根据真实上下文补齐最终方案、当前状态、验证方式、踩坑边界和接手说明。

## 阶段性交付包必须包含

- 甲方原始问题 / 需求；
- 阶段性验收结论；
- 最终有效方案；
- 当前项目状态；
- 本阶段新增能力；
- 人工复现 / 操作步骤；
- 验证方式，并区分智能体可执行验证和人类验证；
- 踩坑点 / 边界；
- 交接给下一团队的说明；
- 后续建议。

## 接棒入口要求

`docs/HANDOFF.md` 使用渐进式披露：

1. 先说明项目定位、当前阶段、目录结构和接棒契约；
2. 列出已完成阶段性交付包；
3. 列出当前待解决问题；
4. 提供按需阅读索引；
5. 不要求新团队一开始读完所有材料。

## 验证责任

智能体应执行：

- 单元测试；
- 脚本 smoke test；
- 静态检查；
- 自己可稳定操作的端到端验证。

交给人类：

- 前端交互；
- 浏览器操作；
- 视觉观感；
- 真实业务判断；
- 账号授权流程；
- 任何人类明显更好操作或更有判断力的验证。

不要硬塞给智能体。
